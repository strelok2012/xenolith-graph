# ADR-0005: Mutable graph storage with read-only public API

**Status:** Accepted
**Date:** 2026-05-22

## Context

The graph model (`Graph`, `Node`, `Edge`, `Comment`) is the most-touched data structure in the library. On a 230-node / 700-edge XXL fixture, a single drag gesture mutates positions at 60 fps for many seconds; the renderer reads the same state every frame to compose the PIXI scene; plugins (undo, serialize, collab) read it from outside. Choice of storage strategy determines both runtime performance and the API every downstream layer sees.

Three candidates were considered.

### Mutable storage

Plain `Map<NodeId, Node>` mutated in place. Mutations are O(1) with zero allocations; renderer subscribes via the event bus and updates exactly the affected sprite. Downside: nothing prevents an outside caller from mutating internal state and bypassing the command log.

### Fully immutable (Immer / structural sharing)

Every mutation returns a new `Graph`. Old states retained by reference for free time-travel. Downsides: per-frame drag of 50 nodes produces thousands of small allocations per second (~3000 GC events under sustained interaction); renderer must diff old vs new each frame, or we replicate the event bus anyway; introduces a runtime dependency (Immer ~14 kB gzip) which violates [ADR-0003](./0003-no-deps-in-core.md).

### Hybrid — mutable internals, read-only external API

Internally, plain mutable Maps. Externally, every public method returns `Readonly<T>` and iterators over the underlying collections. Mutations live behind `@internal` methods that the CommandBus is the only intended caller of. A monotonic `version` counter increments on each mutation so external subscribers (React adapter, signal libraries) can detect change by integer comparison without deep diffing.

This pattern is what Figma, tldraw, Excalidraw, and Yjs/Loro all use under the hood. Fully immutable is favoured for low-mutation-rate documents (text editors, JSON forms) — not for interactive canvas editors.

## Decision

Adopt the **hybrid mutable-storage + read-only API** model.

### Storage rules

- Internal collections use `Map<Id, Entity>` with `Entity` shaped as plain object literals.
- Mutator methods are prefixed `_` and marked `@internal`. They are invoked only from CommandBus-applied commands. There is no runtime guard against external callers in v0.x — convention plus TypeScript `@internal` plus the absence from public type exports is the protection.
- Public read methods return either `Readonly<T>` or `IterableIterator<Readonly<T>>`. Internal Maps are never exposed.
- Every mutation increments `graph.version` (monotonic, never wraps in practice — JS safe integer is ample).
- Mutation events flow through the core `EventEmitter` (see [ADR-0003](./0003-no-deps-in-core.md)). Renderer subscribes; React adapter subscribes via `useSyncExternalStore` driven by `version`.

### Identity rules

- All identifiers are **branded string types** (`NodeId`, `EdgeId`, `PinId`, `CommentId`, `TypeId`). The brand is TS-only and erased at runtime, so the strings remain JSON-serializable verbatim.
- IDs are generated by `createNodeId()` / `createEdgeId()` / `createPinId()` / `createCommentId()` factory functions backed by **UUID v7**. v7 is time-sortable, so string compare matches creation order — useful for stable test output and future CRDT keys.

### What this does not yet decide

- Pin storage layout (in-Node array vs side index). Deferred to the cycle that introduces TypeRegistry.
- Cascade semantics on `_removeNode` (does it also remove attached edges, or does the caller handle it). Deferred to the CommandBus cycle — `RemoveNode` command will be responsible for the cascade explicitly.
- Persistence format. The internal mutable Maps are not the file format. `serialize()` will produce a plain JSON object; the file format lives in `@xenolith/plugin-serialize`.

## Consequences

- The core `Graph` API is small and read-mostly. Misuse is hard.
- Drag performance is bounded by PIXI sprite-update cost, not by garbage collection.
- Undo and time-travel come from the CommandBus log, not from structural sharing.
- A future React/Vue/Svelte adapter is straightforward — subscribe to the bus, expose `version` for memoization.
- A future Yjs/Loro adapter is straightforward — wrap mutator methods, proxy operations into the CRDT doc.

## Reconsider if

- A real-world benchmark demonstrates an ergonomic or correctness win from immutable storage that we can't reproduce with the hybrid plus careful event design.
- The CommandBus protocol grows complex enough that the "trust convention" of `@internal` becomes a maintenance problem. At that point we add a runtime guard (mutator methods accept a `CommandToken` only the bus can mint).
